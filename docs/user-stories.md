# User Stories

## 1. 學習者

### 帳號與初次設定

- 作為學習者，我想註冊並登入帳號，讓我的進度能跨裝置同步。
- 作為學習者，我想選擇目前程度與目標程度，讓系統安排合適內容。
- 作為學習者，我想設定每日學習時間與目標，讓首頁顯示可完成的今日任務。
- 作為學習者，我想在未登入時只看到公開頁面，避免誤以為進度已保存。

驗收重點：

- 未登入使用者不可進入主要 App。
- 初次設定完成前，使用者會被導向 onboarding。
- 所有設定欄位需通過前端與後端 Zod 驗證。

### 課程學習

- 作為學習者，我想在課程地圖中看到 B1、B2、C1、C2 的路徑與完成度。
- 作為學習者，我想進入單元與課堂，查看學習目標、先備技能、文法與單字標籤。
- 作為學習者，我想完成不同固定題型並得到立即評分。
- 作為學習者，我想在離線時閱讀已下載課程並完成固定題型，等恢復連線後同步。

驗收重點：

- 課程結構固定為 Level -> Course -> Unit -> Lesson -> Activity -> Exercise。
- 每堂 Lesson 只包含一至三個主要學習目標。
- 外部資料載入需有 loading、empty、error、retry 狀態。

### 錯誤解釋與複習

- 作為學習者，我想看到錯誤類型、嚴重程度、繁體中文說明與正確例句。
- 作為學習者，我想系統依錯誤自動建立複習項目。
- 作為學習者，我想看到今日到期複習，並在完成後更新掌握度。

驗收重點：

- ErrorRecord 需連結 user、exercise、skill、grammar topic、lesson 與 review queue。
- 複習時間由程式規則決定，不由 AI 決定。
- 答對不必然直接視為完全掌握。

### 寫作與 AI 批改

- 作為學習者，我想提交 B1-C2 各程度支援的寫作任務。
- 作為學習者，我想先看到錯誤位置、類型與修改建議，而不是立即得到整篇改寫。
- 作為學習者，我想第二次提交後再得到完整參考版本。
- 作為學習者，我想保存不同版本並比較修改前後。

驗收重點：

- WritingSubmission 不覆蓋原文。
- WritingVersion 保存每次提交與 AI 回饋。
- AI 輸出必須通過 JSON Schema 與 Zod 驗證。

### 聽力與口說

- 作為學習者，我想正常速度與慢速播放音訊，並可重複播放。
- 作為學習者，我想在聽寫後看到逐字差異。
- 作為學習者，我想錄音、取得轉錄、比較目標句與轉錄差異。
- 作為學習者，我想知道語速、停頓、流暢度與疑似發音問題，但不被宣稱為精準發音分數。

驗收重點：

- 音訊需有逐字稿與授權來源。
- 拒絕錄音權限時需有替代流程。
- 語音資料需有保存與刪除政策。

## 2. 內容編輯者

- 作為內容編輯者，我想建立課程、單元、課堂、活動與題目草稿。
- 作為內容編輯者，我想維護單字、文法主題、例句與音訊。
- 作為內容編輯者，我想要求 AI 生成題目草稿，再送人工審核。
- 作為內容編輯者，我想查看題目錯誤率，判斷是否需要修正內容。

驗收重點：

- 編輯者不可直接發布內容。
- AI 生成內容狀態預設為 draft 或 pending_review。
- 所有內容變更需產生 ContentVersion。

## 3. 審核者

- 作為審核者，我想檢查題目、答案、CEFR 程度與中文解釋。
- 作為審核者，我想核准、拒絕或標記內容錯誤。
- 作為審核者，我想檢查 AI 生成內容是否符合程度與安全規則。

驗收重點：

- 只有 approved 內容可進入 published 流程。
- 審核紀錄需保存 reviewer、狀態、原因與時間。

## 4. 系統管理員

- 作為系統管理員，我想管理角色、功能旗標、發布狀態與安全日誌。
- 作為系統管理員，我想查看每日與每月 AI 成本。
- 作為系統管理員，我想停用有問題的功能或內容。

驗收重點：

- 管理動作必須寫入 audit_logs。
- 管理 API 必須檢查 role 與 RLS。
- 日誌不得保存密碼、token 或完整敏感內容。
